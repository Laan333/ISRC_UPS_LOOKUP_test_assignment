# ISRC / UPC Lookup Aggregator — декомпозиция задачи

Основание: [zadanie.md](zadanie.md). Цель: агрегатор метаданных по **ISRC** и **UPC/EAN** с несколькими публичными источниками, единым JSON и устойчивостью к сбоям провайдеров.

**Инфраструктура (договорённости проекта):** контейнеризация через **Docker обязательна**. **PostgreSQL** и **Redis** подключаются **только если это обоснованно задачей и архитектурой** (оригинальное ТЗ БД не требует — не вводить «ради галочки»; при отказе явно зафиксировать в README и в Technical notes).

## Фаза 1. Контракт API и модели данных

- [x] **GET** `/lookup/isrc/{code}` — валидация формата ISRC (длина, допустимые символы), 422 при явно неверном вводе.
- [x] **GET** `/lookup/upc/{code}` — валидация UPC/EAN (8/12/13 цифр, при необходимости контрольная цифра).
- [x] Pydantic-схемы ответа: `query`, список `providers[]` (`provider`, `found`, `title`, `artist`, при необходимости `album`/`label`, `error`, `raw`), блок `summary` (`found_in`, `confidence`, опционально пояснения).
- [x] Версия/обоснование структуры JSON: если отличается от примера в задании — кратко описать в README.
- [x] Health/readiness: минимум **GET** `/health` (для Docker healthcheck; при наличии БД/Redis — при необходимости **GET** `/ready` с проверкой соединений или упрощённая схема без лишних зависимостей в liveness).

## Фаза 3. Абстракция провайдера и реализация

- [x] Общий интерфейс: `lookup_isrc(code) -> ProviderResult`, `lookup_upc(code) -> ProviderResult` (или один метод с типом запроса).

## Фаза 4. Оркестрация, нормализация, summary

- [x] Параллельные запросы: `asyncio.gather` + подавление исключений per-provider → в результате `error`, не падение всего запроса.
- [x] Нормализация строк: trim, объединение пробелов, единый регистр для сравнения (без потери оригинала в `raw` где нужно).
- [x] **Summary**: `found_in` — число провайдеров с `found=true`; **confidence** — правила (например: high при согласии ≥2, medium при одном, low при несовпадении имён и т.д.) — описать в README.
- [ ] Опционально: лёгкая «склейка» лучшего кандидата для отображения (без обязательного поля в задании — только если улучшает читаемость API).

## Фаза 5. Надёжность и эксплуатация

- [x] Таймауты **httpx** на клиент и на запрос.
- [ ] Отдельный лимит для **playwright**-сессий (не сделано: браузерный стек не подключён).

- [x] Опционально: **retries** только для идемпотентных GET и ограниченно (не маскировать постоянные 404).

- [x] Кэш повторяющихся кодов: **Redis** (если поднят в compose) с TTL **или** in-memory/TTL в одном процессе — trade-off (много инстансов vs простота) в README.
- [x] **Rate limiting**: при необходимости через Redis/sliding window **или** middleware без Redis для single-instance — не маскировать злоупотребление внешними API провайдеров.
